# Sumo-Guard Entity-Relationship (ER) Diagram

This document describes the data structure for the Sumo-Guard health tracking application. Since the project uses Firebase Firestore (NoSQL), the entities are structured as a root `users` collection with hierarchical sub-collections.

## Data Schema Details

### 1. User Entity (`/users/{uid}`)
The core user profile. The Document ID matches the Firebase Authentication UID.
*   **uid**: Unique identifier from Firebase Auth.
*   **onboardingDone**: Flag to determine if the user has completed the initial health setup.

### 2. Sleep Log Entity (`/users/{uid}/sleepLogs/{date}`)
Daily sleep records. Documents are keyed by date string (YYYY-MM-DD) for O(1) retrieval of a specific day's data.
*   **durationHours**: Calculated difference between bedtime and wakeTime.
*   **quality**: User-reported perception of sleep.

### 3. Habit Log Entity (`/users/{uid}/habitLogs/{date}`)
Daily lifestyle metrics. Keyed by date string (YYYY-MM-DD).
*   **stressLevel**: Critical input for cardiovascular and mental health predictions.
*   **smokingCigarettes / alcoholDrinks**: Key variables in long-term disease risk weighting.

### 4. Health Profile Entity (`/users/{uid}/user_health_details/main`)
A singleton document per user containing static/slow-moving health data.
*   **existingConditions / familyHistory**: Array of strings used by the prediction engine to adjust baseline risk percentages.
*   **BMI**: Dynamically calculated in the engine using `weight` and `height`.
